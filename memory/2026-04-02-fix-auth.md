# 2026-04-02 - Fixed Frontend Authentication Issues

## Problem Identified

The viewer page was showing 401 Unauthorized errors because:

1. **Direct fetch calls bypassing RTK Query**: Components like `GaussianViewer.tsx`, `tileManager.service.ts`, and others were making direct `fetch()` calls instead of using RTK Query's baseApi
2. **Wrong base URL**: Direct fetch calls used relative URLs like `/api/v1/...` which went to frontend server (5173) instead of backend (8000)
3. **Wrong token source**: Direct fetch calls tried to get token from `localStorage.getItem('token')` but token is actually stored in Redux persist

## Solution Implemented

### 1. Created API Client Utility (`frontend/src/utils/apiClient.ts`)

Created a centralized utility that:
- Gets token from Redux store (not localStorage)
- Uses correct base URL from environment variables
- Provides `authenticatedFetch()` and `authenticatedFetchJson()` wrappers
- Automatically adds Authorization header to all requests

### 2. Fixed All Components with Direct Fetch Calls

Updated the following files to use the new API client:

**Core Viewer Components:**
- `frontend/src/components/GaussianViewer.tsx` - Fixed camera-config, tiles, BIM elements, and overlays endpoints
- `frontend/src/services/tileManager.service.ts` - No changes needed (uses callback function)

**Hooks:**
- `frontend/src/hooks/useAnnotations.ts` - Fixed all CRUD operations for annotations
- `frontend/src/hooks/useSceneComparison.ts` - Fixed scene metadata fetching

**UI Components:**
- `frontend/src/components/PhotoInspector.tsx` - Fixed photo details fetching
- `frontend/src/components/OrthophotoControls.tsx` - Fixed opacity, visibility, layer order, and upload
- `frontend/src/components/DifferenceVisualization.tsx` - Fixed comparison metrics fetching
- `frontend/src/components/AdaptiveRenderer.tsx` - Fixed render session creation and deletion
- `frontend/src/components/SceneComparison.tsx` - Fixed scene metadata fetching

### 3. Key Changes Made

**Before:**
```typescript
const response = await fetch(`/api/v1/scenes/${sceneId}/camera-config`, {
  headers: {
    Authorization: `Bearer ${localStorage.getItem('token')}`,
  },
});
```

**After:**
```typescript
import { authenticatedFetchJson } from '../utils/apiClient';

const config = await authenticatedFetchJson(`scenes/${sceneId}/camera-config`);
```

## Benefits

1. **Correct URL**: All requests now go to `http://10.0.0.65:8000/api/v1` instead of `http://10.0.0.65:5173/api/v1`
2. **Correct token**: Token is retrieved from Redux store where it's actually stored
3. **Centralized auth logic**: All authentication logic in one place, easier to maintain
4. **Consistent error handling**: All API calls use the same error handling pattern
5. **Type safety**: TypeScript support for JSON responses

## Root Cause Analysis

The issue occurred because:
1. RTK Query was properly configured with baseApi and token handling
2. But some components were written to make direct fetch calls (likely for performance or specific use cases)
3. These direct calls didn't have access to the RTK Query configuration
4. Frontend .env was updated but direct fetch calls didn't use it

## Prevention

Going forward:
- Always use RTK Query endpoints for API calls when possible
- If direct fetch is needed, use the `authenticatedFetch` utility
- Never use `localStorage.getItem('token')` - token is in Redux persist
- Never use relative URLs for API calls - use the utility functions
